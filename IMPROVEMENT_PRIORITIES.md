# 개선 우선순위 정리

이 문서는 현재 `auto_coin_bot` 저장소를 기준으로, 실제 코드 상태를 보고 판단한 `보강해야 할 부분`과 `강화해야 할 부분`을 우선순위별로 정리한 메모입니다.

## 현재 판단

- 이 프로젝트는 `로그/운영 관측성`과 `실전 운영 편의성`이 강한 편입니다.
- 반면 `오프라인 검증`, `재시작 후 상태 복원`, `자동화된 품질 보증`은 아직 약합니다.
- 따라서 다음 단계는 전략을 더 복잡하게 늘리는 것보다, `검증 가능한 구조`를 먼저 만드는 쪽이 맞습니다.

## 보강해야 할 부분

### P0. 백테스트/리플레이 계층

- 현재 구조는 로그를 보고 수동 조정하는 흐름이 강합니다.
- 앞으로는 로컬 OHLCV 데이터를 넣고 전략을 다시 재생할 수 있어야 합니다.
- 1차 목표
  - 알트 MA 전략, BTC EMA 전략을 오프라인에서 재생
  - 거래 이유, 손익, 승률, 최대 낙폭을 요약
  - 결과를 `reports/backtests/...` 아래에 남기기
- 이번 작업에서 `backtest_replay.py` 를 추가해 첫 기반을 만들었습니다.

### P0. 재시작 후 포지션 상태 복원

- 현재는 기존 보유 물량이 감지되면 실제 평균 매수가 대신 현재가를 임시 진입가로 둡니다.
- 이 상태에서는 재기동 직후 손절, 익절, 순익 보호 판단이 왜곡될 수 있습니다.
- 다음 단계
  - `trade_history.jsonl` 기반 평균 진입가 복원
  - 최근 체결과 현재 잔고를 대조하는 포지션 복구 로직 추가
  - 복구 실패 시 거래 중단 + 경고 알림

### P0. 일일 손실 제한의 영속화

- 현재 일일 손익 제한은 메모리 변수로만 유지됩니다.
- 프로세스를 재시작하면 사실상 일일 제한이 초기화됩니다.
- 다음 단계
  - 당일 체결 로그 기준 손익 재계산
  - 또는 별도 상태 파일로 일일 손익 누적 저장

### P1. 자동 테스트와 회귀 방지

- 전략 분기가 많아 수동 검증만으로는 누락이 생기기 쉽습니다.
- 최소 단위 테스트가 필요한 영역
  - SMA/EMA/ATR 계산
  - 거래량/변동성 필터
  - 최소 주문 금액 차단
  - 부분 익절/부분 손절
  - 수수료 반영 순손익 계산
- 이후 CI까지 연결해 회귀를 줄이는 것이 좋습니다.

### P1. 실제 리스크 기반 포지션 사이징

- 현재 `risk_per_trade` 는 이름과 달리 실제 손절 거리 기반 위험금액 계산보다는 `가용 현금 사용 비율`에 가깝습니다.
- 다음 단계
  - `허용 손실 금액 / 손절 거리` 기반 수량 계산
  - 거래소별 최소 주문 단위 반영
  - 심볼별 상한과 거래소 전체 노출 상한 분리

### P1. 공통 로직 분리

- 현재 OKX/업비트 공통 helper 일부가 전략 파일 안에 함께 있습니다.
- 같은 전략 구조를 유지하려는 현재 원칙과도 조금 충돌합니다.
- 다음 단계
  - `market_indicators.py`
  - `okx_exchange_utils.py`
  - `upbit_exchange_utils.py`
  - `exit_rules.py`

### P2. 보안 운영 가이드

- `.env` 는 git 에서 제외되어 있지만, 운영 보안 가이드는 더 보강할 여지가 있습니다.
- 다음 단계
  - 거래 전용 권한만 허용
  - 출금 권한 차단
  - IP 화이트리스트 권장
  - 키 교체 절차 문서화

### P2. 저장소 위생

- `build/`, `dist/` 같은 산출물이 저장소에 포함되어 있습니다.
- 배포 산출물과 소스 이력을 분리하면 리뷰와 변경 추적이 더 쉬워집니다.

## 강화해야 할 부분

### 1. 구조화 로그 체계

- 현재 강점은 `system / strategy / trade` 분리 로그입니다.
- 이 구조는 유지하고, 오히려 백테스트 결과도 같은 관점으로 요약되게 연결하는 것이 좋습니다.
- 목표
  - 실거래 로그와 백테스트 결과의 비교 가능성 확보
  - 전략 버전별 성과 비교 자동화

### 2. 거래 품질 분석

- `MFE`, `MAE`, `slippage`, `fill_ratio`, `api_latency_ms` 를 이미 남기고 있는 점은 매우 좋습니다.
- 이 강점은 앞으로 주문 방식 튜닝과 종목 선별에 더 적극적으로 써야 합니다.
- 목표
  - 종목별 체결 품질 랭킹
  - 거래소별 주문 방식 비교
  - 시간대별 품질 편차 분석

### 3. 운영 복구 흐름

- 텔레그램 인시던트와 재기동 버튼 흐름은 운영 도구로서 충분히 강합니다.
- 앞으로는 여기에 `자동 차단`과 `자동 복구 후 검증`을 얹는 방향이 좋습니다.
- 목표
  - 연속 오류 N회 시 자동 거래 중지
  - 재기동 후 상태 점검 실패 시 추가 경고

### 4. 거래소 간 공통 전략 구조

- AGENTS 원칙대로 OKX 봇과 업비트 봇의 전략 구조를 최대한 맞추는 방향은 계속 유지하는 게 좋습니다.
- 지금은 이미 공통 `.env` 와 공통 전략 로더가 있으니, 다음은 공통 계산 함수와 공통 리플레이 엔진까지 맞추는 단계입니다.

### 5. 저시드 운영 설계

- 최소 주문 금액, 수수료, 먼지잔고, 과매매 방지에 신경 쓴 점은 이 프로젝트의 현실적인 강점입니다.
- 백테스트도 반드시 같은 제약을 반영해야 실전성과 연결됩니다.

## 현재 우선순위 결론

다음 순서가 가장 현실적입니다.

1. 백테스트/리플레이 계층 추가
2. 재시작 후 포지션 상태 복원
3. 일일 손실 제한 영속화
4. 자동 테스트 추가
5. 실제 리스크 기반 포지션 사이징
6. 공통 로직 분리
